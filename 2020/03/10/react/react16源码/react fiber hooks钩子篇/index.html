react fiber hooks钩子篇

发表于 2020-03-10 | 分类于 react |

在 hooks 机制上，fiber reconciler 同样抽象了几种数据类型以承载状态变更的逻辑：Update、UpdateQueue、Hook、Effect 等。以 useState 说明其实现流程如下：

  1. 使用 renderWithHooks 封装组件渲染函数，每次执行期间重置 workInProgressFiber.memoizedState（memoizedState 以链表的形式存储 hook 钩子）
  2. 首次渲染时，渲染函数内部调用 useState，走 mountState 流程，借助 mountWorkInProgressHook 函数创建 hook 实例，并添加到 memoizedState 队列中，构造 dispatch 方法，以数组形式返回 hook.memoizedState 初始状态、dispatch 状态更新方法
  3. 消费 state 进行渲染，workInProgressFiber 转变为 currentFiber（因此 currentFiber.memoizedState 中存储着组件渲染函数所使用的全量钩子），结束 mount 流程
  4. 当已变更 state、再次渲染时，首先重置 workInProgressFiber.memoizedState，此时 currentFiber.memoizedState 依旧存储着全量钩子，因此可基于 currentFiber.memoizedState 区分挂载和更新流程。对于挂载或更新流程，切换不同的 ReactCurrentDispatcher 实现，如 HooksDispatcherOnMount、HooksDispatcherOnUpdate
  5. 再次渲染时，渲染函数内部调用 useState，走 updateState 流程，以此完成状态变更
  6. 消费 state 进行渲染，结束 update 流程

通过上述流程，我们也能感知到，其中最关键的处理在于 renderWithHooks 内部根据函数组件的 mount、update 阶段切换 ReactCurrentDispatcher 实现，借此使 useState 在首次渲染、再次渲染时有不同表现。这里不吝贴出 renderWithHooks 的删减版源码：

renderWithHooks 执行过程会刷新下述变量：

  • currentlyRenderingFiber：标识当前处理中的 fiber。renderWithHooks 执行中置为 workInProgressFiber，执行完置为 null

状态变更

hooks 中的状态变更与 redux 相仿，即采用 reducer(prevState, action) 计算 nextState。新的状态计算产生后，fiber reconciler 会调用 scheduleWork 重新调度当前 fiber 及其子树的渲染任务。这是 state 变更的主流程，由 dispatchAction 函数完成。接下来我们通过这个主流程理解下各种数据结构表现。

Update

  • expirationTime：过期时间，即 deadline 时间，未到该时间点，更新将不予执行
  • suspenseConfig：suspense 配置
  • action：状态变更的规则
  • eagerReducer：暂存上一个 reducer 状态变更逻辑
  • eagerState：暂存上一个状态，以便复用
  • next：指向下一个 update
  • priority：优先级，开发环境有效，通过 getCurrentPriorityLevel 获得，即 scheduler 模块透出的 currentPriority。如react scheduler 再解析篇提到的，unstable_next、unstable_runWithPriority 接口可以变更 currentPriority，从而影响 update 执行

  • pending：存储 update 链表，意义在于使用调度机制支持同步多次更新 state

  • lastRenderedReducer：存储上一个 reducer 状态变更逻辑
  • lastRenderedState：存储上一个 state
