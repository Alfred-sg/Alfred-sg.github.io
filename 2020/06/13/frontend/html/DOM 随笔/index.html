DOM 随笔

2020-06-13
analyst, js

写在篇首

浅近地看，面向对象、面向过程对应着两种思维模式，前者描述事物，后者描述变化。类比我们自小接触的数学方程、力学计算，自变量 x 或物体的质量 m 都是不可再分割的单一属性客体，我们原本就亲近面向过程的思维模式。虽然简单地把小车假想成一个质点可以描述它的运动状态，但是想要打造一辆小车，那就不止于表述小车（系统与环境）的外部表现，还要理清小车（系统）的内部特征。面向对象本身趋近于外在事物在我们头脑中呈现出来的形态。我们会说，“黄鹂拥有飞翔的能力”。事物既包含外在于人的客观实体，又包含由人发明的观念变体（计算机科学中的种种抽象也在此例）。上述两种思维模式为我们提供了认知、编程的工具，在某种程度上又什么都没有提供，为其填充内容的是我们对事物的见解。如果我们空泛地聊着业务两个字，那等于我们聊的不是业务，而是一种自身发展或经他人点拨的价值认同。大多数程序员更多地是通过编程手段将外部事物转化为程序实现，而不是植根于计算机科学去推动技术的发展。在这种带有仿真（或翻译）性质的工作中，被仿真（或翻译）的主客观对象比如汽车工程师所熟知的汽车内部构造、电商pd所熟知的供应链管理体系。因为事物有在环境中自生长的力量，所以当它们被移植到程序世界时，参与者除了要解决一些技术问题外，还需要有足够的预见性和推动力。

上述文字来自于自我表达的需要，就其意义，充其量只是整理一些方向性认识，以便为今后的行动提供一定的指导。可想而知，在战略和战术、规划和实现等等两种高低维度层次中，高维度富于远见，能为低维度的努力纠正方向性的偏差，却不如低维度富于务实气息。高低维度的深化认知也需要相应的职务层级发挥价值。如果方向没错、契机就位，落在较低维度锤炼技艺是相当有福的一件事。直觉地看，业务的发展规模会限制技术实现的形态。发展规模小，需求变化快，就不能指望技术以一种更具远见的方式实现；发展规模大，需求变化快，就会逼迫技术探寻一种一劳永逸的解决方案。简单地说，我所关注的是——是否需要反复投入人力去解决本质上相同的问题，显然这个问题的答案取决于带有阶段性特质的上层结构。说到结构，很多时候，我觉得一般应用开发所面对的是需要施展整合手段的结构性问题，而不是探求新思路、新设想的技术性问题，受到考验的更多是领域知识的完备度、丰富性，手法的娴熟度、精确性。当然，这样设想会反映出我的认知背后施与影响的际遇，这种认知本身会受到极大的挑战。坦诚地说，我自身在编程领域的发展阶段是在不确定中寻找确定，思绪因而流动不息，显得不够具体而微。用简单地话表达这同一个带有哲学意味的反思是，我需要职务环境赋予我一个可以长期耕耘的点。

三种意识

理解一件事的上位意识是认识它的来踪去脉。它从哪里来，意味着它继承了什么，需要解决同类的什么问题；它往哪里去，意味着它还有什么问题，在同类中有哪些不足。这种意识本身以发展的眼光对待这件事；持有这种意识的人也会聚焦于推动这件事的发展。因此这种意识也可以称为建设者的意识。比如玉伯在《做一个简单自由有爱的技术人》中提到语雀的最初致力于解决Confluence文档与组织结构强关联这个缺点；比如 umi 团队在 umi-ui 中画下的蓝图。

理解一件事的下位意识是认为它的形态是约定俗成的，不用去理解它为什么是这样的，只需要知道它现在是怎样的。持有这种意识的人会奉行拿来主义，他们也不会过多占有对这件事的解释权。因此这种意识也可以称为使用者的意识。如果有平级意识，我会把使用过程中的反馈作为人对这件事保有能动性的证明，当然这种能动性不像主要推动者那样具有主导性。美其名，这种意识可以称为贡献者的意识。

我感受到的一种困境是，在下位待得太久了，人总想上一上位。上位意识毕竟体现着创造力，回馈心流，主动性也不至于闭塞；将上位认知应用到下位环境也会有无可比拟的掌控力。理性地看，在知识体系的层级结构中，每一层除了有居于下位按部就班的空间外，也有居于上位开拓进取的空间。使用者也会创造使建设者感到惊讶的玩法，个别会流入建设者的识海，以补充他们考虑的不完美处。

DOM 漫步

面对 DOM，从上位的角度，我可以设问：

  • 为什么 DOM 抽象为树形结构？
  • 浮动、定位、flex 布局的渲染实现？

从下位的角度，我可以设问：

  • 怎么提升 Node 节点增删改性能？

这两类问题都不再关心接口的具体形式，这方面可以参阅文档对象模型 (DOM)。有个模糊并不缜密的意识提醒我，除了难以捉摸的设计外，其他都可以被工具书吸纳且逻辑乏味。

说到 DOM 树，我先捎带一个源于经验的感性认识。datav 组件、photoshop 图层等使用了平铺结构。但是在使用过程中，树形的目录结构更便于管理。在布局调整上，datav、photoshop 能灵活地摆布部件；宜搭限制了组件的布局，常规是一行一个组件，借助特殊的容器能实现一行多个组件。拿应用场景加以比较，datav、photoshop 均针对视觉内容多变的界面；宜搭针对视觉泛化的管理端界面。作为两种类型的可视化编辑器，因为业务需求的不同，datav、宜搭实现上存在种种差异。如果想把可视化编辑器中的视觉内容转译成代码，不带层级的平铺结构必然使 html 不够直观，同时也会增加样式的编码量。也许这就是 DOM 构成树形结构的直观原因。

技术上，html 经由词法分析、语法分析转换为 DOM 树（html 中开闭标签构成一个完整节点的过程使用）；DOM 节点因其共性构成逐级而下的继承关系，因其特性会有不同的节点类型。在 Node -> Element -> HTMLElement 的层级结构中，Node 级持有父子节点、兄弟节点的引用，并抽象了节点的增删改操作；Element 级抽象了查询节点、设置属性、绑定事件等操作；HTMLElement 级抽象了配置样式等操作。这些可以被理解，却不能站在建设者的视角道出个它们的来由去向。

有时候对直觉予以肯定也是有益的。空泛地讲，学养内化为直觉、自信或气质，直觉依托于归纳、演绎等一般性技巧，驰骋在材料、具体问题上，这段经验反过来构成直觉的养料。因此有三种自我提升手段：培养直觉、锻炼技巧、收集材料。具体地讲，当我为实现某个功能去寻找调用接口时，直觉告诉我，如果找不到那样的接口，那就说明这个功能太个性了，或者提供接口的平台还不够成熟。浏览器还不够成熟吗？这个说法也是在重申，需求导致了技术上的变革，似乎理解了需求就理解了实现；除了设计，仿佛其他都是枝节。

事物是逐步发展的，揭示它发展的轨线好比预示潮流，理解它在某个时间点的剖面好比追逐潮流。我们看到，DOM 1 级包含了 CORE、HTML 模块；DOM 2 级包含了 Style、Events 等模块；DOM 3/4 级扩展了 Events 模块，出于支持键盘、触屏事件的需要。可以认为，样式和事件都是因为现实需要而后来实现的。实际上，在样式出现之前，设计者们使用不同的元素来呈现不同的样式。选择器不只支持 js 脚本查询和操作节点，同时又支持层叠样式表中通过选择器为元素设置样式。因为列表中存在多个同质元素，伪类选择器也就有了存在的理由。与此同时，针对不同的选择器，也就需要设置权限以支持差别对待。

技术上，DOM 树会先剔除如 meta 等纯语义节点，构成 RenderObject 树。RenderObject 节点会通过排版引擎计算出元素的样式布局信息。我们知道，元素布局时采用盒模型，呈现出块形特征。在层级上，每个元素有其从属于的包含块，比如块状和内联元素被包含在父级块中，块状元素又向下构成子孙元素的包含块；绝对定位的元素被包含在相对定位的祖先元素下。通过需求理解实现的角度看，块状和内联元素分别提供了垂直、水平排列的能力。浮动和定位技术的出场来自于满足从右侧起始和遮罩等布局的需要；flex 布局是出于满足同质元素横向或纵向排列的需要。定位着眼于一个；flex 着眼于多个。在技术上，通过递归计算子孙元素的位置、大小等信息，包含块的布局才得以确定；HTMLElement 中也有元素距离文档的位置信息。

浏览器还会基于 RenderObject 树创建 RenderLayer 树，RenderLayer 树是渲染引擎的主要处理对象。有了 RenderLayer，针对各层可以采用不同的渲染方式，如采用了 3D 绘图上下文对象的 canvas 元素。对于 RenderLayer 子层，可以作细微的处理，如透明的元素、或设置 z-index 的元素、或溢出的元素。渲染引擎基于 RenderLayer 树，将 RenderObject 的样式信息绘制为显示内容，每一层构成一个图像，这些图像又通过合成技术合并为同一张图像。《Webkit 技术内幕》点明了单层 RenderLayer 中 RenderObject 子树的渲染过程：递归绘制所有子孙元素的背景；顺序绘制 z-index 为负数的子孙元素；绘制浮动的子孙元素；绘制子孙元素的内容和轮廓；顺序处理子孙元素的 overflow 属性；顺序绘制 z-index 为正数的子孙元素；处理该元素的滤镜效果。

以上两段说的是样式的呈现过程，接下来看的是网页中的事件机制。先插一番感性的推理，Node 节点的树形结构无疑为事件捕获、冒泡提供了可能，那为什么要提供事件冒泡呢？或者 Node 节点采用平铺结构，当有了事件冒泡的需求时，会发生什么呢？按演化的视角看待事物的发展，事物的终极形态并不是一蹴而就的。新的需要为事物的演化提供了契机。一旦事物无法以革新的方式实现这个需要时，而这个需要在权衡利弊的过程中又不得不实现，那它所面临的的唯一结果就是推倒重来。因为这个原因，淘宝从单体应用转向了 HSF 分布式服务；路易十六被架上了断头台。后面那个事例意在说明，程序员提供的技术方案有时候就跟英国的君主立宪制、法国的民主共和制一样，都是为了解决结构性问题，在施行过程中也会遇到各自的技术问题，无问优劣。

技术上，Node 节点继承了 EventTarget，因此通过 Node 可以绑定、解绑事件处理函数，以及分发事件。当事件发生时，渲染引擎会通过算法向下捕获查找事件目标节点，找到目标节点后，事件又会向上传递。事件处理函数可以设置为在冒泡阶段触发或在捕获阶段触发，默认为冒泡阶段。从需求理解实现的角度，阻止事件冒泡行为和组织事件的默认处理行为都是可以被理解的。

无论节点操作类，还是 mvvm 框架，都需要对 Node 节点增删改查、样式和事件等作进一步封装。jQuery 等节点操作类库针对浏览器的差异性实现提供了简单便利的调用接口，可是它跟 JDBC 一样，繁琐编码方式既不直观，又不便于维护。handlebars 等模板引擎有助于轻松糅合界面展示及数据实体，抽象的数据实体和形象的页面展示也是分离的，便于组合，但是在事件绑定方面仍旧显得低效且繁琐，例如需要查询完节点后再为其绑定事件，数据变更后需要手动调用渲染函数实现刷新。这两个问题揭示了技术演进方向：提供新的模板引擎，能将局部作用域中的方法及属性注入到模板中；作用域中的数据实现双向绑定，值变更能自动反馈为局部视图的刷新。前者意味着 mvvm 框架会为真实的 dom 节点完成事件绑定，后者意味着框架会更细腻地调用节点操作接口。

在 mvvm 框架中，首先通过组件形式经纬数据、方法以及模板，它可以构成上下文，因此在模板渲染过程中能为其注入数据、事件处理函数；随后虚拟 dom 作为模板渲染的结果，将与真实的 dom 进行对接，一旦虚拟 dom 中的属性发生了变更，就会调用 DOM 接口以增量更新的形式只重绘对应的元素。在渲染性能方面，增量更新适合重绘单元格，批量更新适合重绘整行。但是，批量更新需要重新生成子 DOM 树并计算节点的布局，也有一定性能损耗。mvvm 框架有两种处理：使用深度优先算法逐层对比虚拟 dom 逐层的属性变更情况，当且仅当数据变更时重绘对应的元素，这样可以提升单元格的重绘性能；虚拟 dom 的 key 键可以提升整行重绘的性能。
