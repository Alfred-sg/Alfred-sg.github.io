babel 处理流程管窥

2020-07-09
analyst, js

原理

babel 处理三步骤：parse 解析、transform 转换、generate 生成。

  • 解析过程基于 @babel/parser 做词法分析、语法分析。词法分析将源程序解析成令牌流；语法分析将令牌流转换成 AST。
  • 转换过程基于 @babel/traverse 遍历 AST，对节点进行添加、更新及移除等操作。此时插件将介入工作。
  • 生成过程基于 @babel/generator 深度遍历 AST，将其转换成字符串形式的代码，并生成 source maps 源码映射。

AST

babel 抽象语法树格式 基于 ESTree 规范。对于 jsx 语法，AST 又基于 Facebook JSX AST。AST Explorer 直观地呈现了源程序会转变成怎样的 AST 语法树及相应的 json 表现。AST 节点模型及 Program 节点如下：

解析

@babel/parser 源于 acorn。acorn 为插件架构，@babel/parser 内部也是插件架构，这样可以支持 es、flow、jsx、ts 语法。其插件机制，由 EstreeParser 等子类扩展 Parser 父类实现，即通过不同的子类重写方法体。在主处理流程方面，@babel/parser 分为两步：

  1. 通过 Tokenizer 将源程序字符串解析成令牌流
  2. 通过 Parser 构建 AST 抽象语法树

令牌指 Token 实例，它的属性包含 type 类型（如 num、string 等）、value 具体的数值或字符串、以及位置信息等。令牌流的解析流程为：

  1. 通过 State#pos 标记当前处理的源程序字符串索引
  2. 逐个读取字符，根据首字符类型作策略处理。比如当首字符为 ‘.’，则预读下一个字符以区分小数、’…’、’.’。字符类型基于 charcodes 判断。详见 tokenizer#getTokenFromCode 方法
  3. 根据字符流类型创建 Token 令牌。记录令牌信息的方式有两种：通过 state.type、state.value 记录，或通过 state.tokens 记录

Tokenizer 对外提供了逐个取词的能力：

  • next：解析令牌流，存储在 state.tokens 中。
  • nextToken：解析一个令牌，存储在 state.type, state.value 中。
  • eat(type)：解析一个令牌。若令牌类型与 type 匹配，返回 true；否则返回 false。

单以 if (expr) stmt 语句为例，按上下文无关文法，它最终会形成 IfStatement 节点，以及其下的 SequenceExpression、Statement 节点。语义分析最终施诸于程序实现，因此 @babel/parser 实现了 parseIfStatement、parseExpression、parseStatement 方法。对于每种类型的 Node 节点，其构建过程为，@babel/parser 首先会使用 startNode、startNodeAt 等方法创建 Node 实例及其初始位置信息，然后在 parseXXX 过程中为该 Node 实例添加属性，如 IfStatement#test 条件语句、IfStatement#consequent 为执行体，最后调用 finishNode 方法为 Node 实例设置类型和结束位置信息。在此过程中，因为 Node 节点的嵌套层级，parseXXX 方法会呈现递归调用的特点；语句 Statement 也有多态特征，parseStatement 等方法中也有多个条件语句分支，其中一个是 parseIfStatement。与此同时，一些辅助性的标识如 ‘(‘、’)’ 不构成节点，会在程序中被剔除。通过如下代码，我们可以对其流程管窥一二：

转换

@babel/traverse 用于遍历 AST 树。

为了新增、删除、更新节点，babel 使用 NodePath 路径操纵节点。值得一提的是，为了拆分、组装多个功能模块，@babel/parser 中的 Parser 采用不断继承的方式；NodePath 则将多个功能模块导出的方法归并到原型对象中。NodePath 的功能模块包含增删改查节点等操作，变更会反应到 AST 抽象语法树上，最终影响生成的代码。对于 NodePath，@babel/traverse 通过访问者模式进行遍历，babel 插件以此实现它们的功能。如下演示了对节点进行访问并改写的操作：

@babel/traverse 遍历 AST 以生成 NodePath 及访问 Node 节点的处理流程主要有：

  1. 构建 TraversalContext 实例。该实例可视为上下文环境，其 scope 属性可视为作用域
  2. 针对每种节点类型，调用 TraversalContext#visit 方法 进行访问。TraversalContext#visit 在 AST 树遍历过程中会递归调用，因此它会对每种节点类型、多棵子树等情况作出处理
  3. TraversalContext#visit 执行过程中，首先 AST 树中的 Node 节点会被转换成 NodePath，其次将 Node 节点的信息会同步给 NodePath，其次为 NodePath 创建新的作用域或复用父级作用域，最后使用 NodePath#visit 对其进行访问操作
  4. NodePath#visit 执行过程中，会逐步调用外置的 enter 访问方法，其次再递归访问子节点，最后调用外置的 exit 访问方法

其他模块

  • @babel/types：适用于操纵 AST 树的类 lodash 工具库。功能包含类型检查、定义节点类型、生成节点等，详见文档。
  • @babel/template：输入字符串，基于 @babel/types 生成的节点替换字符串中的占位符，以制作新的 AST 树。
