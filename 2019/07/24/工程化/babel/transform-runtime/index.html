transform-runtime

发表于 2019-07-24 | 分类于 前端工程化 |

@babel/plugin-transform-runtime 插件用于在编译阶段为工程添加垫片。plugin-transform-runtime 插件与 @babel/polyfill, core-js 的差别是：避免多次导入；提供沙箱环境，避免全局污染。.babelrc 配置可如下：

基于以上配置，babel 就会在编译项目时，主动为工程文件装填 @babel/runtime-corejs2（一般安装在构建工具中）。当使用裸跑的 webpack 构建项目时，则可以同时安装 @babel/plugin-transform-runtime, @babel/runtime-corejs2，然后再配置 .babelrc 文件。

plugin-transform-runtime 包含选项如下：

  • corejs: 指定垫片，值可以是 false, 2, 3，分别为 @babel/runtime, @babel/runtime-corejs2, @babel/runtime-corejs3 垫片。
  • helpers: classCallCheck 等函数是否内置，还是导入 ‘@babel/runtime/helpers/classCallCheck’ 等模块。默认 true，内置。
  • regenerator: 是否以污染全局命名空间的方式支持生成器语法，还是导入 ‘@babel/runtime/regenerator’。默认 true，污染全局命名空间。
  • useESModules: 是否通过 @babel/plugin-transform-modules-commonjs 编译成 commonjs 语法。默认 false，编译成 commonjs。useESModules 置为 true 时，适用于 webpack 模块系统，因为 webpack 不需要保留 commonjs 语法。
  • absoluteRuntime: 默认为 false。设置值时，将从指定的文件中查找 runtime 垫片。

需要特别说明的是，npm 和 cnpm 安装依赖的方式有所不同，npm 将子类库安装在父类库的 node_modules 文件夹下，cnpm 将子类库安装在子类库的 node_modules 文件夹下。absoluteRuntime 选项又基于工程文件查找 corejs 模块。因此当构建工具中的 webpack 选项通过内嵌的子类库设置时，实际的 corejs 模块路径会跟着 npm 或 cnpm 安装方式有所不同，处理上可以相对于 require.resolve(‘@babel/plugin-transform-runtime’) 查找。
