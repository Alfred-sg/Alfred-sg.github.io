我看 spring boot

发表于 2019-12-07 | 分类于 java |

why & what

spring boot 应运而生的原因在于 spring 应用配置相对复杂，其目的在于简化 spring 项目中依赖的配置流程。它包含如下特性：

  • SpringApplication：提供一种简便的方式启动 spring 应用。
  • Externalized Configuration：允许使用外部配置启动 spring 应用，通过 spring.profile.active 区分不同环境；并允许在应用中访问这些配置项。
  • Web Applications：spring boot 集成了 spring web mvc。它内嵌 Tomcat、Jetty、Undertow、Netty 服务器，便于一键启动；它也支持使用 spring-boot-starter-webflux 模块启动响应式 web 应用。
  • SQL：提供使用 SQL 数据库的广泛支持，从 JDBC 到对象关系映射框架（如 Hibernate、Mybatis）等。
  • NoSQL：支持 Redis、MongoDB、Neo4j、Elasticssearch、Solr Cassandra、Couchbase、LDAP 等 NoSQL 数据库。
  • Caching：spring boot 允许在方法上使用 @Cachable，该注解将根据请求参数作缓存。默认使用内存缓存，也支持 Generic、JCache、EhCache、Hazelcast、Infinispan、Couchbase、Redis、Caffeine、Simple 等缓存服务（spring boot 会自动检测）。
  • Message：支持 JMS、RabbitMQ、Kafka、WebSocket 等消息系统。
  • Rest Services：spring boot 自动装配了 RestTemplateBuilder，凭此可以创建 RestTemplate 实例，然后调用 rest 服务；同样的，WebClientBuilder 可用于创建 WebClient 实例，然后调用 rest 服务。
  • Validation：借助于 bean 校验机制 JSR-303 实现（如 Hibernate validator），spring boot 允许使用 @Validated 注解对 bean 作校验。
  • Logging：集成日志系统，便于调试和监控应用。
  • Testing：提供一系列实用工具和注解，以帮助开发者测试应用。
  • Auto Configuration：自动配置以简化配置流程、定制各种特定 bean 的能力。
  • Monitoring：监控，更好地掌握应用的运行状态。

SpringApplication

为了简化 spring 的配置流程，首先需要劫持 spring 的启动过程，spring boot 中的 SpringApplication 就是用于启动 spring 应用（通过显式调用 SpringApplication.run 方法）。在应用过程中，FailureAnalyzer 将分析启动失败的原因并打印在日志中（允许定制 FailureAnalyzer 的实现）；如果没有错误提示日志，可以尝试启用 debug 模式启用日志。SpringApplication 包含但不限于如下功能点：

  • 在 spring 机制的基础上，spring boot 支持 bean 的懒初始化（即在请求到达时才创建 bean），这样可以节省应用的启动时间。缺点是没法在启动过程中发现 bean 的配置问题，也没法保证运行时创建的 bean 会使 jvm 内存不足。懒初始化默认禁用，可以使用 java 编码或 lazy-initialization 配置启动。
  • 允许通过 SpringApplicationBuilder 创建多个 ApplicationContext 实例（构成父子结构的层级关系），环境变量会在 ApplicationContext 实例中共享，web 组件只能运行在子 ApplicationContext 实例上。
  • spring boot 基于以下机制自动判断 ApplicationContext 的类型：如果使用了 Spring MVC，就使用 AnnotationConfigServletWebServerApplicationContext；如果使用了 Spring WebFlux，就使用 AnnotationConfigReactiveWebServerApplicationContext；以上情况之外（比如同时使用 spring MVC 和 spring WebFlux），就使用 AnnotationConfigApplicationContext。可以使用硬编码的方式改用 ApplicationContext 的实现类。
  • 在 spring 事件机制的基础上，spring boot 允许通过 SpringApplication.addListeners、SpringApplicationBuilder.listeners 或设置 META-INF/spring.factories 方式添加应用启动过程的事件监听器。子 ApplicationContext 实例的事件会冒泡到父级，需要区分 ApplicationContext。一般不使用事件。
  • 允许使用 ApplicationArguments 接口访问启动参数 args。

Externalized Configuration

spring boot 支持以下形式的外部配置：properties 文件、yaml 文件、环境变量、命令行参数。有了这些外部配置，就能实现在不同应用公用一份配置的能力。这些配置也能通过 @Value 注入到 bean 中；或通过 @ConfigurationProperties 绑定到结构化对象上。外部配置抽象为 PropertySource，按以下顺序加载：

  1. $HOME/.config/spring-boot 文件下的 devtool 全局配置。
  2. 在测试文件中使用 @TestPropertySource 加载的配置。
  3. @SpringBootTest 或其他测试注解加载的配置。
  4. 命令行参数。
  5. 通过环境变量或系统变量设置的 SPRING_APPLICATION_JSON 配置。
  6. ServletConfig 初始化参数。
  7. ServletContext 初始化参数。
  8. 从 java:comp/env 获得的 JNDI 属性。
  9. 可以通过 System.getProperties 获得 java 系统属性。
  10. 操作系统环境变量。
  11. 通过 random.* 设置的随机值 RandomValuePropertySource。
  12. jar 包外指定 profile 环境配置信息的 properties、yaml 文件。通过 –spring.profiles.active=dev 指定 profile；通过 spring.profiles.active 访问当前的 profile；当前激活的 profile 可以通过 spring.profiles = default 指定，前缀 ! 表示取反；配置文件内可以使用 ${variable} 复用配置项；yaml 文件可以使用 YamlPropertiesFactoryBean 解析成 properties，使用 YamlMapFactoryBean 解析成 map；YamlPropertySourceLoader 能将 yaml 配置导出到 spring 环境中，随后才能使用 @Value 注解引用。
  13. jar 包内指定 profile 环境配置信息的 properties、yaml 文件。
  14. jar 包外不含 profile 环境配置信息的 properties、yaml 文件。
  15. jar 包内不含 profile 环境配置信息的 properties、yaml 文件。
  16. @Configuration 配置类中的 @PropertySource 注解。
  17. 通过 SpringApplication.setDefaultProperties 设置的默认配置。

Web Applications

spring boot 集成了 spring web mvc，它将自动注册 Converter、GenericConverter、Formatter；使用 HttpMessageConverter 接口转化请求和响应，如将对象转化成 json；支持通过 JsonSerializer、JsonDeserializer 超类转化 json 数据；支持使用 FreeMarker、Groovy、Thymeleaf、Mustache 等模板引擎；默认发送 /static 或 /public 或 /resources 或 /META-INF/resources 文件夹下的静态资源，可以通过定制 WebMvcConfigurer 中的 addResourceHandlers 改写静态资源位置；允许通过 WebMvcConfigurer 提供 cors 跨域访问能力等。可参考 文档。

Logging

spring boot 默认使用 Commons Logging 打印内部日志，支持通过配置形式切换成使用 Java Util Logging、Log4J2、Logback。默认情况下，如果使用了 “Starters”，就会使用 Logback 记录日志，允许通过 Logback 路由使独立库使用其他的日志系统（且可以独立设置日志级别）。日志会通过控制台输出，也可以记录为文件形式，通过 logging.file.name、logging.file.path 可以指定日志输出文件。默认日志包含 ERROR、WARN、INFO，通过命令行参数 –debug 或 debug=true 配置可以额外打印 DEBUG 级别日志；命令行参数 –trace 允许额外打印追踪级别日志。

繁多的日志系统可使用指定路径下的配置文件，也可以使用 logging.config 配置项更改配置文件的路径；可以使用 org.springframework.boot.logging.LoggingSystem 环境属性指定使用某个日志系统。
