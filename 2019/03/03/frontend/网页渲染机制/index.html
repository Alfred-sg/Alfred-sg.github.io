网页渲染机制

按特定的理解方式，当用户在浏览器地址栏输入 url 起，网页渲染过程包含页面加载和渲染两个过程。网页加载过程指从服务器端请求资源到构建 DOM 树的过程。网页渲染过程指从 DOM 树渲染除视图内容。这两个过程有所重合，因为异步加载的 js 脚本可能会修改 DOM 树的结构。

在 webkit 中，浏览器会构建多个加载类加载不同类型的资源，大体策略为：先从本地缓存池中查找资源；如果找不到，再从网络中获取资源。这些资源以 url 作为唯一标识符。浏览器中有两种缓存，内存缓存和磁盘缓存：刷新网页将使用内存缓存以提升速度；否则从磁盘读取。网页资源通常是同步加载；对于 javascript，采用异步加载的形式，且会打断 DOM 树的构建。

浏览器首先加载网页内容，使用 html 解释器将网页转变成一系列的 token，再根据 token 构建 DOM 树。在一个可见的 DOM 节点（不包含 head 元素和 display 置为 none 的节点）插入到 DOM 树的同时，浏览器同时会构建 RenderObject 树，该树中包含节点的样式信息等内容，且使用布局模块（排版引擎）计算节点的大小位置等布局信息。在 css 2 中，RenderObject 节点都称为 box，包含的属性有：width, height, margin， padding, left, top, border 等。当网页中存在 css 样式时，首先将使用 css 解释器计算各元素的样式信息，然后将样式信息插入到 RenderObject 树中。当网页中存在 javascript 样式时，首先使用 javascript 引擎解释脚本，并通过 DOM 接口和 CSSOM 接口来修改网页内容和样式信息，最终也会影响 RenderObject 树。

为处理 position 定位等，浏览器在 RenderObject 树之外，还会构建 RenderLayer 树。比如根节点从属于一个 RenderLayer 节点，绝对定位的节点从属于另一个 RenderLayer 节点，这样就能有效地处理视图元素的层次结构。如果某 DOM 节点没有对应的 RenderLayer，就从属于父节点的 RenderLayer。

在有了 RenderObject 树和 RenderLayer 树的基础上，渲染引擎首先以一个布局信息不够明确的矩形绘制 RenderLayer 节点，再遍历其下的子 RenderLayer 节点并完成相关的 RenderObject 节点的绘制，再向上递归调整 RenderLayer 节点的布局（即回流）。因此，RenderObject 树决定网页的内容，RenderLayer 树决定网页的层次结构。RenderLayer 布局由 CPU 处理，然后将位图作为纹理上传给 GPU，由 GPU 绘制内容并缓存（如果下次上传的渲染层没有发生变化，GPU 就不需要对其进行重绘）。chrome 浏览器的开发者工具 - more tools - Layers、rendering 插件可用于查看网页层级及渲染性能。

Navigation Timing

上图简要地讲述了网页渲染过程中触发的事件，从查询 DNS 到建立 TCP 连接、到请求资源、到文档就绪、到所有资源加载完成等的一整个过程。各事件说明如下：

  • navigationStart 前一份文档卸载完成的时间戳。如果没有前一份文档，则返回 fetchStart
  • redirectStart 如果发生了 HTTP 重定向，且重定向和当前文档同域，则返回重定向的 fetchStart。否则，返回 0
  • unloadEventStart 前一份文档的 unload 事件触发时间
  • unloadEventEnd 前一份文档的 unload 事件执行完成时间
  • redirectEnd 如果发生了 HTTP 重定向，且重定向和当前文档同域，则返回最后一次重定向接受完数据的时间。否则，返回 0
  • fetchStart 浏览器发起资源请求时，如果有缓存，则返回读取缓存的开始时间。否则，返回获取资源的时间
  • domainLookupStart 查询 DNS 的开始时间。如果请求没有发起 DNS 请求，如 keep-alive，缓存等，则返回 fetchStart
  • domainLookupEnd 查询 DNS 的结束时间。如果没有发起 DNS 请求，同上
  • connectStart 开始建立 TCP 连接的时间。如果请求是 keep-alive，缓存等，则返回 domainLookupEnd
  • secureConnectionStart 如果是 https 请求，则返回握手时间。否则，返回 0
  • connectEnd 完成 TCP 连接的时间。如果是 keep-alive，缓存等，同 connectStart
  • requestStart 发起请求的时间，包含请求服务器资源、本地缓存等
  • responseStart 服务器或本地缓存开始响应的时间
  • responseEnd 服务器或本地缓存结束响应的时间
  • domLoading 文档对象 document 已创建，文档状态被置为 ‘loading’ 前的时间
  • domInteractive 文档状态被置为 ‘interactive’ 前的时间
  • domContentLoadedEventStart 文档对象触发 ‘DOMContentLoaded’ 事件前的时间
  • domContentLoadedEventEnd 文档对象触发 ‘DOMContentLoaded’ 事件后的时间
  • domComplete 文档状态被置为 ‘complete’ 前的时间
  • loadEventStart 文档的 load 事件触发时间，如没有则返回 0
  • loadEventEnd 文档的 load 事件执行完成时间，如没有则返回 0

各事件说明：

  • Queuing 请求排队中。在 HTTP 1 上，浏览器仅允许每个源拥有六个 TCP 连接；若请求的资源为低优先级资源，将让位给高优先级的样式或脚本资源
  • Stalled 请求等待发送所用的时间，包含代理协商所用的任何时间
  • DNS Lookup 执行 DNS 查询所用的时间。页面上的每一个新域都需要完整的往返才能执行 DNS 查询
  • Initial Connection 建立连接所用的时间，包括 TCP 握手/重试和协商 SSL 的时间
  • SSL 完成 SSL 握手所用的时间
  • Request Sent 发出网络请求所用的时间
  • Waiting(TTFB) 等待初始响应所用的时间，也称为至第一字节的响应时间
  • Content Download 接收响应数据所用的时间

performance

使用 Performance API 可以分析网页的渲染性能以及某个方法的执行性能。

performance.timing 属性可追踪的网页渲染性能如下：

  • DNS 寻址时间：domainLookupEnd - domainLookupStart
  • TCP 连接耗时：connectEnd - connectStart
  • request 请求耗时：responseEnd - requestStart
  • 白屏时间：responseStart - navigationStart
  • 解析 dom 树耗时：domComplete - domInteractive
  • domready 时间(用户可操作时间节点)：domContentLoadedEventEnd - navigationStart
  • onload 时间(总下载时间)：loadEventEnd - navigationStart

performance 接口包含的方法如下：

  • performance.getEntries()：每个资源的请求时间
  • performance.now()：获取从 navigationStart 到当前时间的毫秒，可用于计算方法的执行时间
  • performance.mark()：给相应的视点做标记
  • performance.measure(name, startMark, endMark)：计算方法的执行时间
  • performance.getEntries(name)：获取资源的加载时间

回流、重绘机制

回流（重排）指的是，当元素的布局或显示等信息改变时，引起 RenderObject 树部分或整体的重新构建，即网页布局的调整。重绘指的是，RenderObject 树中节点属性的更新。因此，回流必然会引起重绘，重绘不一定引起回流。与布局无关的元素属性操作将只引起重绘，否则将同时引起回流和重绘。可能引起回流、重绘的行为如下：

  • 添加、删除元素（回流+重绘）
  • 隐藏元素：display:none（回流+重绘）；visibility:hidden（只重绘，不回流）
  • 移动元素：改变 top,left（不一定引起回流）；将元素移动到另一个父元素中（回流+重绘）等
  • 改变 style：布局相关属性如 padding, border-width, font-size（回流+重绘）；布局无关属性（只重绘，不回流）。可查看 https://csstriggers.com/ 以获取更多信息
  • 用户操作：改变浏览器大小，改变浏览器的字体大小等（回流+重绘）

回流的开销比重绘大。如果 body 顶部插入一个节点，将引起整个 body 的回流；如果尾部插入一个节点，将只引起部分内容的回流，因此开销较小。浏览器为提升回流、重绘的性能，会构建队列以执行批量处理。但是，访问节点的 offsetTop, scrollTop, clientTop, width, height 类属性或调用了 getComputedStyle 方法，将迫使浏览器提前执行队列以获得最新的样式。减少回流、重绘的方法如下：

  1. className 或 cssText 批量更新样式，避免单属性操作引起的频繁回流或重绘
  2. 新创建的元素改完样式后，再插入文档；或者以 display 为 none 形式调整样式
  3. 使用变量缓存元素的样式，避免频繁读取元素的样式，致使浏览器提前执行回流队列
  4. 指定图片的宽高，避免新加载的图片调整大小时引起的回流
  5. [慎用]使频繁回流、重绘的元素单独有一个 RenderLayer：借助 video 元素、WebGL、Canvas、CSS3 3D、CSS滤镜、z-index 大于某个相邻节点的元素都会有独立的 RenderLayer，比如通过添加 transform: translateZ(0);
    backface-visibility: hidden; 样式即可。

回流、重置可借助 dynaTrace(测试ie),Speed Tracer(测试Chrome) 测试。
