scheduler 源码分析

发表于 2019-01-15 | 分类于 react |

requestHostCallback

针对不同平台构建 requestHostCallback, cancelHostCallback, shouldYieldToHost, getCurrentTime 函数。用于执行回调，基于 rAF 实现的回调执行策略也将与平台的重绘机制契合。特别的，基于 rAF 实现的上述四个函数，requestHostCallback 用于保证回调（或可理解为任务）在重绘后执行，cancelHostCallback 用于取消任务，shouldYieldToHost 用于判断任务是否超时、需要被打断，getCurrentTime 用于获取当前时间。

关于 getCurrentTime 函数的实现：如果平台支持 performance，那么 getCurrentTime 函数的返回值就是 performance.now()；否则返回 Date.now()。在设置 _schedMock 的前提下，getCurrentTime 由 _schedMock 提供。

基于 _schedMock

如果设置了 window._schedMock 或 global._schedMock 数组，上述四种函数就相应设置为指定的数组项。

基于 setTimeout

当没有设置 _schedMock，且 window 是 undefined 或 MessageChannel 的类型不是 function 时，React 将基于 setTimeout, clearTimeout 定时器实现上述函数。该实现的过程中，首先以 _callback 变量缓存回调，_flushCallback 函数一次性调用 _callback 回调；其次按下述步骤实现指定函数。

  1. requestHostCallback(cb, ms): 分两次调用 setTimeout 制作定时器：第一次使用 requestHostCallback 自身作为回调；第二次，调用前先将 cb 缓存到 _callback 中，然后使用定时器执行 _flushCallback，由 _flushCallback 间接调用实际的入参 cb。
  2. cancelHostCallback: 清除缓存 _callback。
  3. shouldYieldToHost: 实现为返回 true 的函数。

基于 rAF

当没有设置 _schedMock，且 window 不是 undefined 或 MessageChannel 的类型是 function 时，React 将基于 requestAnimationFrame, cancelAnimationFrame 实现上述函数。

该实现中包含如下缓存：

  • scheduledHostCallback 意为待执行的回调。我们可以把它理解成任务，下文也将用任务指代。因为 scheduler 使用 rAF 轮询任务，因此当任务先后添加时，在一个重绘周期内所添加并执行的任务流可视为队列。
  • timeoutTime 记录回调执行允许的超时时间，值为时间点，而非时长。
  • isAnimationFrameScheduled 真值时意味着 rAF 轮询已启动。通过 requestHostCallback 函数启动轮询，在 rAF 执行回调 animationTick 时若 scheduledHostCallback 为 null，则退出 rAF 轮询。rAF 轮询的意义在于保证任务的执行时机为重绘完成后。
  • isMessageEventScheduled 真值时意味着 onMessage 监听函数正在执行过程中。
  • isFlushingHostCallback 真值时意味着某个任务正在执行中。这时，scheduler 允许在该任务的执行期间，直接启动另一个任务的执行逻辑；或者说，isFlushingHostCallback 对于其他任务来说现在是可执行的（requestHostCallback 函数基于此调用 postMessage 发送消息，以执行刚添加的任务）。
  • activeFrameTime 作为一帧的理想执行时长，预设为 33 ms。当页面刷新率很高时，这个值也会在 animationTick 执行过程中被更新，即变小。
  • previousFrameTime 前一帧执行时长。
  • frameDeadline 一帧执行完成的期望时间点。当我们的代码不包含视图更新时，setTimeout 将取代 rAF 唤醒回调（或者说没有空闲），造成 onMessage 监听函数的执行时机大于 frameDeadline。在这种情况下，frameDeadline 可用于回调是否在默许的超时范围内，onmessage 监听函数就是这么做的。

以下是实现过程：

一、实现 requestAnimationFrameWithTimeout(callback) 函数。该函数存在的意义为，当我们的代码不包含视图更新时，requestAnimationFrame 将得不到调用，因此作为兜底，scheduler 在 requestAnimationFrame 之外，还使用 setTimeout 调用 callback，以免 callback 没有执行。介于 requestAnimationFrame, setTimeout 都将 callback 作为回调，为避免两次调用，callback 内势必要借助某种机制避免第二次调用的有效执行。以下是 requestAnimationFrameWithTimeout 的实现。

二、实现 requestHostCallback(callback, absoluteTimeout)。首先将 scheduledHostCallback 更新为参数 callback，timeoutTime 更新为 absoluteTimeout。若 isFlushingHostCallback 为真或者 absoluteTimeout 超时时间为否值，使用 MessageChanel 机制发送消息，以执行 callback。否则，在 isAnimationFrameScheduled 为否值（即 rAF 轮询未启动的情况下），scheduler 将启动 rAF 轮询。

三、实现 animationTick 函数。该函数作为 rAF 的回调，也即 requestAnimationFrameWithTimeout 的参数。其处理流程为：

  1. 如果 scheduledHostCallback 不是 null，持续启用 rAF 轮询新添加的任务（以便在一个重绘周期内批处理多个任务）；否则将 isAnimationFrameScheduled 置为 false，并中断 rAF 轮询。
  2. 帧率调整、下一帧执行完成的期望时间点计算。包含如下步骤：
    • 以当前帧执行完成的实际时间点减去期望时间点、再加上理想执行时长的方式，计算 nextFrameTime。
    • 若前一帧和当前帧的执行时长均小于理想执行时长，将理想执行时长 activeFrameTime 调整为这两帧执行时长的最大值；若这两帧执行时长有一个大于理想执行时长，将前一帧的执行时长 previousFrameTime 缓存更新为当前帧的执行时长。
    • 以当前帧执行完成的实际时间点加上理想执行时长的方式，计算下一帧执行完成的期望时间点 frameDeadline。
  3. 当 isMessageEventScheduled 为 false 时，将 isMessageEventScheduled 置为 true，同时调用 postMessage 发送消息。

四、实现 channel.port1.onmessage 监听函数，作为实际执行任务的触发器。其处理逻辑为：

  1. 将 isMessageEventScheduled 置为 false。
  2. 通过对比 frameDeadline 和 onmessage 监听函数的执行时机，判断 setTimeout 是否取代 rAF 唤醒回调，如是，判断超时时间是否在默许的范围内：当超出了默许的范围，以 true 作为首参执行任务；当没有超出，启动 rAF 轮询，在下一次重绘后执行任务。
  3. 将 isFlushingHostCallback 置为 true，并执行任务。在任务执行完后成，再将 isFlushingHostCallback 置为 false。这时如果 scheduledHostCallback 为 null，也会造成 onmessage 监听函空转。

五、实现 cancelHostCallback，取消任务。

六、实现 shouldYieldToHost，通过当前时间超过 frameDeadline 一帧执行完成时间点，判断任务是否超时。如果超时，可用于打断任务的执行，在下一个重绘完成时间点再予以执行。
